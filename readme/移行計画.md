いいですね、その流れかなり相性いいです 👍
やりたいことを整理すると：

1. **Eclipse 依存を外して、Gradle プロジェクト ＋ WebLogic という形にする**
2. **IDE を VS Code に切り替えても同じように開発・デプロイできるようにする**
3. **その完成形の環境を VirtualBox の VM としてチームに配布する**

なので、

* **A. プロジェクトの「IDE依存」分離**
* **B. VS Code ＋ WebLogic 開発フローの構築**
* **C. VirtualBox VM 化と配布**

の3ステップで話します。

---

## A. Eclipse 依存を外す（Gradle＋WebLogicに寄せる）

まずは「Eclipse がいなくても動く状態」にしておくのが先です。

### A-1. Gradle プロジェクトを IDE 非依存に整える

やることはシンプルで：

1. **`build.gradle` / `settings.gradle` をプロジェクトの中心にする**

   * Java / Web アプリなら最低限こんな感じになっていることを確認：

   ```groovy
   plugins {
       id 'java'
       id 'war'
   }

   group = 'com.example'
   version = '1.0.0'

   repositories {
       mavenCentral()
   }

   dependencies {
       providedCompile 'jakarta.servlet:jakarta.servlet-api:4.0.4' // WebLogic のバージョンにあわせて
       // ほか依存
   }

   tasks.withType(JavaCompile) {
       options.encoding = 'UTF-8'
   }
   ```

2. **Gradle Wrapper を用意**

   * Eclipse 関係なく、`./gradlew` だけでビルドできるようにします。

   ```bash
   gradle wrapper
   ```

   → これで `gradlew` / `gradlew.bat` / `gradle/wrapper/` が作成されれば OK。

3. **Eclipse 専用ファイルは「無視してもよい状態」に**

   * `.project`, `.classpath`, `.settings/` はそのまま置いておいても問題なし
   * ただし今後は IDE 設定ではなく Gradle を正とする

---

### A-2. Eclipse 内臓ドメインから「標準 WebLogic ドメイン」へ

Eclipse で WebLogic を動かしているときは、たいてい：

* Eclipse 配下のどこかに **開発用ドメイン** が作られている
* そこにアプリがデプロイされている

ので、今後のために **Eclipse とは別の場所に WebLogic ドメインを作る** のがおすすめです。

1. WebLogic のインストールディレクトリへ移動

2. `config.sh` / `config.cmd` を実行して新しいドメイン作成

   * 例：`mydomain`
   * Admin ポート `7001`、ユーザ名/パスワードなどを設定

3. AdminServer を起動してコンソールに入れることを確認

   * `http://localhost:7001/console`

4. この「独立ドメイン」に対して、後で war をデプロイする想定にします。

---

## B. VS Code ＋ WebLogic 開発フローの構築

Eclipse の「Run on Server」を、
VS Code では **「Gradle でビルド → WebLogic にデプロイ → ブラウザで確認」** という形に変えます。

### B-1. VS Code 側の準備

1. **必要な拡張機能**

   * `Extension Pack for Java`（Java 用フルセット）
   * `Gradle for Java` など（お好み）
   * その他：EditorConfig などはお好み

2. **JDK の指定**

   * VS Code の `settings.json` に `java.home` を設定
   * WebLogic で使用する JDK と合わせておくとトラブル減ります。

3. **プロジェクトを VS Code で開く**

   * `File → Open Folder...` で Gradle プロジェクトのルートを指定
   * Java プロジェクトとして認識されるのを確認

---

### B-2. Gradle でビルド → war 作成

VS Code のターミナルで：

```bash
./gradlew clean build
# または war タスクを明示
./gradlew clean war
```

* `build/libs/xxx.war` ができていれば OK
* まずはここまでを「VS Code 単体で出来る状態」にします。

---

### B-3. WebLogic へのデプロイ方法を決める

開発環境なら、シンプルに次のどれかで OK です。

#### 方法1：管理コンソールから手動デプロイ（最初はこれで）

1. `http://localhost:7001/console` にアクセス
2. **Deployments → Install**
3. `build/libs/xxx.war` を選択
4. ターゲットサーバ（AdminServer or ManagedServer）を選ぶ
5. デプロイ完了後、`http://localhost:7001/アプリコンテキスト` で確認

→ まずはこの「手動運用」で橋渡しをしておくと、
　**Eclipse からの「Run on Server」と同等のことができる状態** になります。

#### 方法2：autodeploy を利用（開発用の簡易運用）

1. WebLogic ドメインの `autodeploy` パスを確認

   * 例）`$DOMAIN_HOME/autodeploy`
2. Gradle の `build` 後に、自動で war をコピーするタスクを作る：

```groovy
tasks.register('deployToWebLogic', Copy) {
    from tasks.war.outputs.files
    into 'C:/Oracle/user_projects/domains/mydomain/autodeploy'
}
```

* 実行フロー：

  1. `./gradlew clean war deployToWebLogic`
  2. WebLogic が autodeploy フォルダを検知して自動デプロイ

→ Eclipse の「保存 → 再デプロイ」ほど細かくはないですが、
**コマンド一発でビルド＆デプロイ**な仕組みになります。

#### 方法3：VS Code の `tasks.json` でワンキー実行

VS Code にタスク定義しておくと：

* `Ctrl+Shift+B` やコマンドパレットから
  **「build + deploy」タスクを実行**できるようになります。

例）`.vscode/tasks.json`（ざっくりイメージ）:

```json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "build & deploy to WebLogic",
      "type": "shell",
      "command": "./gradlew clean war deployToWebLogic",
      "group": "build",
      "problemMatcher": []
    }
  ]
}
```

---

### B-4. WebLogic の起動も VS Code から行いたい場合

`.vscode/tasks.json` にもうひとつタスクを追加して、

```json
{
  "label": "start WebLogic",
  "type": "shell",
  "command": "C:/Oracle/user_projects/domains/mydomain/startWebLogic.cmd",
  "problemMatcher": []
}
```

みたいにしておくと、VS Code から：

1. 「start WebLogic」
2. 「build & deploy to WebLogic」

という流れで実行できて、だいぶ Eclipse に近い体験になります。

---

## C. VirtualBox で環境ごと配布する

ここまでで、**ローカルの VS Code + WebLogic + Gradle 環境**が整いました。
これを VirtualBox の VM として固めて配布するイメージです。

### C-1. VM 内に入れるもの（構成イメージ）

* OS：Windows / Oracle Linux / CentOS / Ubuntu など（チーム事情に合わせる）
* JDK：WebLogic がサポートしているバージョン
* WebLogic 本体＆ドメイン

  * `MW_HOME`, `DOMAIN_HOME`
* VS Code

  * Java 拡張、Gradle、（必要なら Git）
* Git / プロジェクトソース

  * リポジトリをクローン済みにしておくか、clone 手順をドキュメント化
* Gradle Wrapper（プロジェクト内に含まれていれば OK）

### C-2. VM セットアップの流れ

1. **VirtualBox で新規 VM 作成**

   * CPU / メモリは開発用にそこそこ（例：2〜4コア、8GB〜）
2. OS インストール
3. JDK インストール、環境変数設定（`JAVA_HOME`, `PATH`）
4. WebLogic インストール、ドメイン作成（本番は別にあるなら開発用ドメイン）
5. VS Code インストール、必要拡張を追加
6. プロジェクトを clone or コピー
7. **B で作った VS Code タスク＋ Gradle の動作確認**

   * `./gradlew clean build`
   * 「start WebLogic」タスク
   * 「build & deploy」タスク
   * ブラウザで動作確認

ここまでできたら、その VM が **「チーム共通の標準開発環境」** になります。

---

### C-3. チームへの配布方法（OVA エクスポート）

1. VM の状態をきれいに整える

   * 余計なログ削除
   * OSアップデート完了
   * ドキュメント（README）を `/home/dev/README`, `C:\dev\README.txt` などに置く
2. VirtualBox メニュー → **「ファイル → エクスポート」**

   * OVA（Open Virtual Appliance）形式で書き出す
3. OVA ファイルを共有

   * NAS、クラウドストレージ、USB など
4. 各メンバーは VirtualBox で **「インポート」**

   * ネットワーク設定など必要に応じて調整
   * 起動 → README の手順に従えば、
     **全員同じバージョンの WebLogic＋JDK＋VS Code＋Gradle で開発開始**できます

## まとめ

* **今やるべき順序としては：**

1. Gradle プロジェクトを Eclipse から切り離し
   → `./gradlew clean build` でビルドできる状態にする
2. Eclipse とは別に WebLogic ドメインを作って、
   **手動デプロイで動作確認**
3. VS Code でプロジェクトを開き、

   * Gradle ビルド
   * WebLogic 起動タスク
   * build & deploy タスク
     を整える
4. その完成形を VirtualBox VM に乗せて、OVA エクスポートして配布

* Web アプリ？（war）それとも ear 構成？
* OS：Windows で VM 作る前提？ Linux でもOK？
* デプロイは autodeploy 運用でよいか、それともコンソール／WLST まで含めたいか

たとえば

> 「Windows の VirtualBox VM 前提で、war アプリを autodeploy する Gradle 設定と VS Code の tasks.json を全部書いて」
> みたいに指定してもらえれば、そのままコピペして調整できるレベルで書きます。
なるほど、「VM は Windows で作りたいけど、WebLogic を Windows に入れると“評価版扱い”になってしまうのでは？」って不安ですよね。

結論からいうと：

> **OS が Windows か Linux かで「評価版かどうか」が変わるわけではない**
> → WebLogic は **どの OS 上でも、商用利用するならライセンスが必要** です。
> → 逆に言うと、**開発用途なら Windows でも Linux でも「開発用途の範囲」で使う前提** になります。

（※正確なライセンス条件は、最終的には Oracle のライセンス文書を確認 or ベンダーに確認、が必要です）

## ざっくり整理すると

### 1. OS で無料／有料が変わるプロダクトもあるが…

たとえば：

* **Oracle DB XE（Express Edition）**
  → これはエディションそのものが「無償版」で、OS によって扱いが変わるわけではない
* 逆に、**Windows Server 自体のライセンス**は別途必要

なので、

> 「Linux なら本番でも無料だけど、Windows だと評価版になる」

みたいな世界観では *ない* です。

### 2. WebLogic の場合

ざっくりイメージとしては：

* **開発／検証用途**

  * 開発者ライセンスや評価利用の範囲で使う
  * OS は問わない（Windows でも Linux でも同じ）
* **本番運用用途**

  * どの OS であれ、**WebLogic の商用ライセンスが必要**

なので、「Linux なら無料、本番OK」「Windows だから評価版になる」という区別ではありません。

---

## チーム用 VM をどう考えるか

あなたのケースだと目的は：

> 「開発環境（VS Code + Gradle + WebLogic）を VirtualBox VM に固めて、
> チームに配布して **共通の開発環境として使いたい**」

つまり **“本番運用” ではなく “開発用途”** の VM ですね。

この場合のポイントは：

1. **OS は Windows でも Linux でも OK**

   * チームメンバーが慣れている方でよい
   * ライセンス的には WebLogic 自体の利用形態（開発用途か、本番用途か）が問題になる

2. **VM 配布台数分もライセンス的にどうカウントされるか**

   * これは正直、Oracle 製品のライセンスルールに依存します
   * 「開発者ライセンスで何人まで OK か」「リモート開発環境として配る場合の扱い」などは
     → 自社のライセンス契約窓口 or ベンダー経由で確認するのが安全

## 実務的なオススメ

* **技術的な観点だけなら：**

  * VirtualBox のゲスト OS は **Linux (Oracle Linux) + WebLogic** のほうが将来性・情報量も多くてオススメ
  * ただし、チームが Windows に慣れているなら Windows でも問題なし

* **ライセンス的に安全にいきたいなら：**

  * 会社として契約している WebLogic のライセンス種別を確認
  * 「開発環境用 VM を何台まで作ってよいか」を
    ライセンス窓口 or Oracle パートナーに一度聞いておく
